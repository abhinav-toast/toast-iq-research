The Core Shift
Philosophy
Right now, most agent skills are designed around a single question: what should the agent do? The research demands a second question be asked first: has the agent earned the right to do this yet? Every skill needs to carry its own trust scaffolding — not as a wrapper added later, but as part of its anatomy from the start.
Build These First
Infrastructure Skills — Required Before Domain Skills
These are not domain skills. They are the infrastructure that every domain skill depends on. Build them first and compose them in — do not duplicate their logic per skill.
Infra #1
data_health_check
Validates prerequisite data before any recommendation skill surfaces output. Surfaces a DataHealthCard when data is stale, missing, or unverifiable.

Output

Infra #2

Traverses the full entity relationship graph for any priced entity. Required before surfacing any change recommendation — not optional.

The Skill Output Contract
Required Output Schema — Every Skill
Structural Requirement
Why This Matters
Every skill today returns its primary result. Going forward, every skill output must carry four additional fields. These are not optional enhancements — they are load-bearing trust scaffolding. If a field is missing, a required UI component cannot render, and the trust contract breaks.
{ result: <primary output>, provenance: string, // plain-language explanation of data origin basis: string, // the assumption this result rests on — makes it falsifiable affected_entities: Entity[], // downstream items, modifiers, locations impacted rollback_token: string | null // reversibility handle — null for read-only skills }
  • provenance
    P03 asked "where does this data come from?" multiple times. Operators won't act on numbers they can't trace. Provenance is a first-class output, not a tooltip buried in settings.
  • basis
    P03's "$847" problem. An unanchored projection erodes trust. Every projection must expose the assumption it rests on ("based on 847 units sold last month") so operators can sanity-check against what they know.
  • affected_entities
    P04's modifier incident. A pricing change is never just a pricing change. If the skill doesn't map what else it touches, the operator either catches it manually or discovers it in production. Neither builds trust.
  • rollback_token
    Reversibility is a trust signal, not an escape hatch. Its presence tells the operator the agent understands consequences. Skills that modify state must return this; read-only skills return null.
Behavioral Modes — Propose / Draft / Execute
Most Important Structural Change
Every action skill must declare and respect a behavioral mode. No skill should default to execute. Skills that currently go straight to action need to be decomposed into at minimum a propose step and a separate execute step, with an explicit draft state between them. These are not the same skill with a confirmation dialog — they are structurally separate phases with different state ownership.
Propose
Returns a recommendation with full reasoning. No state is created or modified. Operator reads and decides whether to proceed.
Default for all skills until operator configuration explicitly elevates it.
Draft
Creates a persistent, named draft capturing the full change package. Draft survives sessions. Addressable to a reviewer role, not just the initiating user.
Nothing is published until a designated approver acts.
Execute
Applies the change. Only available after a draft has been reviewed and approved.
Should never be reachable in a single step from a cold start.
// Declare in every skill definition mode: "propose" | "draft" | "execute"
9 Structural Changes
1
Skill Output Contract — 4 Required Extra Fields
Schema ▼
Every skill output must carry provenance, basis, affected_entities, and rollback_token alongside its primary result. Without these fields, the required UI components cannot render and the operator trust contract breaks. See the full schema above.
2
Behavioral Modes — Propose / Draft / Execute
Structural ▼
No skill should default to execute. Decompose all action skills into propose → draft → execute as structurally separate phases. See the mode grid above for definitions.
3
Data Health Check — Gate Every Recommendation
Pattern ▼
Before any recommendation skill surfaces its output, it must run a silent data health check. If underlying data is stale, incomplete, or unverifiable, the skill surfaces a DataHealthCard instead of the recommendation — not an error state, a helpful honest card. This is not a separate skill the orchestrator calls; it's a mandatory prerequisite baked into every recommendation skill.
Before
input
→ recommendation
After
input
→ data_health_check
→ [healthy] recommendation
→ [unhealthy] DataHealthCard
    + what's missing
    + how to fix it
Research Evidence
Directly addresses P01's recipe staleness blocker and P03's data provenance skepticism. Operators stop doubting the agent's logic when the agent itself is visibly checking its own inputs before making claims.
4
Cascade Mapping — Required Step in All Change Skills
Structural ▼
Any skill that proposes a change to a priced entity — item, modifier group, category, bundle — must include a cascade map in its output before surfacing the recommendation. The skill needs access to the entity relationship graph (item → modifier groups → modifiers → substitutions). This must be declared as an input dependency.
Before
detect cost spike
→ recommend price increase
→ show confirm
After
detect cost spike
→ map affected modifiers,
    add-ons, bundles
→ calculate cascade impact
→ recommend WITH cascade
→ confirm WITH full cascade
Research Evidence
P04's modifier incident was not an edge case — it was a predictable consequence of treating pricing as item-level when it's actually graph-level. The operator sees the full picture or they don't see the recommendation at all.
5
Stakeholder Model — Initiator ≠ Approver
Schema ▼
Current skills assume the person running the skill approves its output. The research shows this is structurally wrong. P04: GM runs the skill; owners approve. P02: operations manager runs; pricing requires org sign-off. P03: owner initiates but still requires deliberate personal review — not in the same session. Draft skills need a reviewer_role field assigned to a role, not a user. The draft cannot auto-advance until someone in that role acts.
draft: { id: uuid, created_by: user_id, reviewer_role: "owner" | "operator" | "manager", expires_at: null, // persistent by default scope: "this_location" | "all_locations" | "select_locations", status: "pending_review" | "approved" | "rejected" | "modified" }
6
Urgency Language — Configurable, Not Hardcoded
Schema ▼
Skills that surface recommendations should not hardcode urgency framing. "Act now" and "effective tomorrow" are calibrated for high-volatility cost environments — they actively erode trust with QSR operators who change prices quarterly. Skills must return a recommendation_timing object; the urgency label is derived from the intersection of sensitivity and that operator's actual cadence.
recommendation_timing: { sensitivity: "high" | "medium" | "low", operator_cadence: "immediate" | "monthly" | "quarterly" | "annual", suggested_action_window: string, // derived from sensitivity × cadence urgency_label: string // generated, never hardcoded } // Example: high sensitivity + quarterly cadence // → "Consider for your next menu review" not "Act now"
7
Progressive Disclosure — Two-Tier Output Schema
Schema ▼
This is not a UI concern — it's a skill output schema concern. If the skill returns a wall of text, the surface can't progressively disclose it. The structure must come from the skill. Every skill output must be structured in two tiers so any surface can render them appropriately.
output: { summary: { headline: string, // what the skill found or recommends recommendation: string, // primary result in plain language expected_outcome: string // what happens if followed }, detail: { reasoning: string, data_sources: string[], alternatives: Alternative[], affected_entities: Entity[], projection_basis: string } }
8
Scope Awareness — Multi-Location from Invocation
Structural ▼
Scope is not a post-hoc filter — it shapes the entire skill execution. If the skill doesn't support a declared scope, it fails explicitly rather than silently applying to the wrong set of locations. For P02, a skill without scope awareness requires manual repetition across 4 locations — that's worse than no skill at all.
Before
skill runs
→ change proposed
→ operator applies
    manually per location
After
scope declared at invocation
→ skill executes within scope
→ draft is scoped
→ publish is scoped
invocation: { scope: { type: "single" | "all" | "select", location_ids: string[] | null } }
9
Anti-Patterns to Retire
Retire ▼
Six patterns that exist in current skills and must be eliminated. Each one breaks a specific aspect of the operator trust contract.
  • ✕
    Single-step action skills
    Any skill that takes input and directly modifies state without an intermediate draft/review step. → Decompose into propose + draft + execute.
  • ✕
    Unattributed numbers
    Any skill returning a dollar figure, percentage, or projection without basis and provenance. The number is meaningless without its supporting claim.
  • ✕
    Hardcoded urgency copy
    Any skill emitting "act now," "effective immediately," or similar as a static string. → Replace with the configurable urgency pattern (Change #6).
  • ✕
    Initiator-as-approver assumption
    Any skill that resolves its output to the user who ran it, assuming they have authority to approve. → Introduce the reviewer_role model (Change #5).
  • ✕
    Flat output structure
    Any skill returning undifferentiated text or a single result object without summary/detail tiers. → Restructure to two-tier output schema (Change #7).
  • ✕
    Silent cascade
    Any change skill that proposes modifying a priced entity without first mapping modifier and add-on impact. → Add cascade_mapper as a mandatory prerequisite (Change #4).
Core Principle
The skill output contract is not just a data structure — it's a component specification. Every field in the output requires a corresponding UI component to render it correctly. If we know the schema, we know the components. If a component is missing, a skill output field is being discarded or collapsed into plain text, which breaks the trust contract.

5 Composition Rules
  • 1
    Phase gates the available set
    A Phase 1 operator only ever sees Intelligence Components. DraftWorkspace doesn't exist for them — not because it's hidden, but because the skill is running in propose mode and won't generate a draft object until trust has been established. The component set is unlocked by the operator's trust phase, not by feature flags.
  • 2
    Skill output fields determine required components
    If the skill returns affected_entities, a CascadeMap is required — not optional. If the skill returns a projection without a basis, AnchoredProjection should refuse to render an unanchored number and surface a degraded state instead. The component enforces the output contract.
  • 3
    Complexity determines component depth
    A change affecting 2 entities → CascadeMap renders as a compact list. A change affecting 14 entities with 3 arbitrage warnings → CascadeMap renders as an expanded tree with warnings surfaced at the top. The component adapts to the complexity of its input, not a fixed template.
  • 4
    Operator behavior refines defaults
    If the operator has expanded the detail tier of the last 8 RecommendationCards, the component should default to detail-open for this operator. If they've dismissed UrgencyBadges without reading them for 3 months, the agent should de-escalate urgency framing. Component defaults are operator-specific, not universal.
  • 5
    Trust signals are never optional
    ProvenanceLabel, AnchoredProjection basis, and ReversibilityIndicator cannot be removed for space or aesthetic reasons. They are load-bearing. If the layout can't fit them, the layout is wrong.
